LiMem Reimplementation and Design Extension - Next Steps
========================================================

This file outlines practical next steps to:
1) better reimplement the LiMem-style evaluation, and
2) extend the design into a stronger research prototype.


PART A - Better Reimplementation Fidelity
-----------------------------------------

1) Read and map paper details to code
- Extract the exact LiMem definitions used in the paper.
- Confirm denominator choices and any filtering rules.
- Note model settings (temperature, decoding, prompt format).
- Write this mapping in a short "spec" document before coding.

2) Expand puzzle dataset quality
- Grow from 5 pairs to a larger benchmark (for example, 100+ pairs).
- For each original item, create multiple perturbations:
  - name swaps
  - sentence order changes
  - equivalent phrasing changes
  - distractor text that should not change logic
- Keep logic constant across original/perturbed versions.

3) Add data schema and validation
- Store pairs in JSONL/CSV with explicit fields:
  - pair_id, split, original_text, perturbed_text, gold_answer
- Build a validator script to catch:
  - missing pairs
  - invalid labels
  - duplicate IDs
  - accidental logic changes

4) Replace MockLLM with real model calls
- Add a model interface with pluggable backends:
  - local model
  - API model
- Keep run settings fixed for reproducibility.
- Log prompt, response, parse result, and raw output.

5) Improve answer parsing and grading
- Implement strict and robust parsers for model outputs.
- Normalize answer formats (e.g., "A knight, B knave" variants).
- Track parse failures separately from reasoning errors.

6) Add reproducibility controls
- Set random seeds where applicable.
- Save exact run config to a timestamped artifact.
- Record model/version info and environment metadata.

7) Add basic statistical reporting
- Compute confidence intervals (e.g., bootstrap).
- Report results by perturbation type, not just global averages.
- Add per-item error analysis examples.


PART B - Potential Design Additions
-----------------------------------

1) Multi-perturbation robustness score
- For each original puzzle, evaluate across K perturbations.
- Measure variance in correctness over perturbations.
- Add a "stability" metric to complement LiMem.

2) Difficulty-controlled evaluation
- Tag each puzzle by logic complexity (single-step vs multi-step).
- Report LiMem by difficulty bucket.
- Check whether memorization effects increase with complexity.

3) Prompt sensitivity study
- Test multiple prompt templates:
  - direct answer
  - chain-of-thought style instruction
  - constrained answer format
- Compare whether memorization gap changes by prompt style.

4) Cross-model comparison
- Evaluate at least 2-4 models of different sizes/families.
- Compare Acc, CR, LiMem, and error patterns.
- Analyze whether larger models are more stable under perturbation.

5) Human baseline and sanity checks
- Solve a subset manually for label verification.
- Use inter-annotator agreement for uncertain items.
- Build adversarial checks to ensure perturbations stay equivalent.

6) Integrate with project pipeline
- Add scripts:
  - scripts/run_limem_eval.py
  - scripts/summarize_limem_results.py
- Add outputs:
  - results/raw_predictions.jsonl
  - results/metrics_summary.json
- Add one command in README for end-to-end evaluation.

7) Optional extension: training-time mitigation
- Try data augmentation using perturbation pairs during fine-tuning.
- Compare before/after LiMem to test if robustness improves.
- Track trade-off with normal task accuracy.


Suggested Immediate Priority Order
----------------------------------

Week 1:
- Implement real model interface
- Move puzzle pairs into JSONL
- Add parser + grading + logging

Week 2:
- Expand dataset and perturbation coverage
- Add confidence intervals and subgroup analyses
- Produce a short report with key findings

Week 3+:
- Run multi-model comparison
- Test mitigation strategies (augmentation/prompting)
- Finalize reproducible pipeline and documentation
